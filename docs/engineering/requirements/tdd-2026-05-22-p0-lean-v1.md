# TDD: P0 Lean V1

## Status

- Owner: Engineering
- Status: Draft
- Last updated: 2026-05-24

## Purpose

Define the `P0` technical design for the first implementation slice of the lean Weekly Equity Intelligence system.

This document covers the smallest useful system that can:

- ingest low-cost delayed data,
- compute a lean weekly stock ranking,
- persist recommendations and outcomes,
- and support a weekly review loop.

## Scope

This `P0` design is for:

- one user
- batch-first operation
- stock engine only
- `FMP Starter + SEC EDGAR`
- no options pipeline
- no real-time workflows

Temporary bootstrap mode may substitute `yfinance + SEC EDGAR` only for local validation of the signal spine and storage model.

## User-Facing Outcome

The system should produce:

- a weekly ranked review of watchlist stocks
- simple action labels
- a daily change digest
- a stock detail view or report with signal context and recent filings

## P0 Technical Goals

- keep the system operable by one person
- minimize vendor count
- preserve traceability
- support historical recommendation review
- avoid hard-coding vendor payloads into business logic

## P0 Components

### 1. Watchlist store

Stores:

- ticker
- company name
- sector
- strategy tags
- position status
- active / inactive
- created / updated timestamps

### 2. Source adapters

#### FMP adapter

Fetches:

- daily OHLCV
- benchmark and sector prices
- earnings calendar / results if available in starter plan
- core fundamentals if available in starter plan

#### Bootstrap `yfinance` adapter

Fetches:

- daily OHLCV
- adjusted close when available
- lightweight earnings-date snapshots when available

Constraints:

- local or manually supervised use only
- not the canonical long-term source for unattended jobs
- not sufficient for the broader product data contract

#### SEC adapter

Fetches:

- `10-K`, `10-Q`, `8-K` metadata
- filing URLs
- raw filing text or referenced text capture

### 3. Normalized storage

Core normalized tables:

- `securities`
- `watchlist_entries`
- `daily_prices`
- `benchmark_prices`
- `earnings_events`
- `fundamental_periods`
- `filings`
- `weekly_runs`
- `stock_scores`
- `recommendations`

### 4. Derived feature layer

Derived features for each active watchlist name:

- `ma_10`
- `ma_20`
- `ma_50`
- `ma_200`
- `atr_14`
- `price_vs_ma_20`
- `price_vs_ma_50`
- `relative_strength_vs_spy_4w`
- `relative_strength_vs_sector_4w`
- `distance_from_52w_high`
- `volume_ratio_20d`
- `earnings_days_to_next`
- `post_earnings_flag`
- `basic_fundamental_trend_flag`

### 5. Weekly scoring layer

Generates a lean set of outputs:

- trend score
- relative strength score
- event risk flag
- fundamental support flag
- overall action label

### 6. Delivery layer

Initial acceptable delivery forms:

- generated markdown or HTML report
- CLI summary
- simple internal page later

Do not block the first implementation on a polished UI.

## Data Flow

### Daily flow

1. Load active watchlist.
2. Fetch latest stock prices from `FMP`.
3. Fetch benchmark and sector data from `FMP`.
4. Fetch earnings updates and basic fundamentals from `FMP`.
5. Fetch new filings from `SEC`.
6. Normalize and store.
7. Recompute derived features for changed names.
8. Generate daily change digest rows.

### Weekly flow

1. Recompute all active watchlist features.
2. Score each stock.
3. Generate action labels.
4. Persist weekly run metadata.
5. Persist stock-level recommendation records.
6. Publish weekly report.

## Scoring Design

The first implementation should stay intentionally simple.

### Trend score

Inputs:

- price above / below key moving averages
- moving-average slope proxies
- distance from recent highs

### Relative strength score

Inputs:

- 4-week relative performance vs `SPY`
- 4-week relative performance vs sector ETF

### Event risk flag

Inputs:

- days until next earnings
- recent earnings event
- recent material filing

### Fundamental support flag

Inputs:

- basic trend in revenue and margins if available
- debt / cash sanity checks if available

### Output label

One of:

- `Buy candidate`
- `Watch for entry`
- `Hold`
- `Deteriorating`
- `Avoid / no action`

## Storage Strategy

### Raw layer

Store raw payloads or raw extracts for:

- FMP responses used for daily ingestion
- SEC filing metadata and source references

### Normalized layer

Store clean typed records for downstream logic.

### Derived layer

Store computed features and scores with run timestamps.

This separation is mandatory even in `P0`.

## Scheduling Strategy

### Daily job

Run:

- after market close

Responsibilities:

- data ingestion
- feature refresh
- daily digest generation

### Weekly job

Run:

- weekend or Friday after close

Responsibilities:

- full rerank
- weekly report generation
- historical recommendation persistence

## Error Handling

### Hard-fail conditions

- cannot fetch core watchlist price data
- cannot write normalized data
- weekly run cannot complete ranking for most active names

### Soft-fail conditions

- missing filing text for one filing
- missing fundamental field for one company
- temporary benchmark field issue that can be backfilled later

Soft failures should degrade confidence, not crash the full weekly workflow.

## Observability Requirements

Log for each job:

- run start and end time
- records fetched
- records normalized
- failures by endpoint
- changed tickers count
- weekly report generation success

## Security and Secrets

- store API keys in environment variables
- keep secrets out of the repo
- preserve provider usage limits in job design

## Upgrade Hooks

Design `P0` so we can later add:

- `FMP Premium/Ultimate`
- transcripts
- richer revisions
- dedicated options providers
- richer UI delivery

without rewriting canonical tables for:

- securities
- prices
- earnings
- filings
- scores
- recommendations

## Non-Goals

- options schema in `P0`
- transcript summarization pipeline in `P0`
- real-time event handling
- multi-provider consensus logic
- optimization-heavy backtesting engine

## Acceptance Criteria

`P0` is complete when:

- active watchlist ingestion runs daily without manual intervention
- weekly rerank completes on the full watchlist
- weekly report is generated and saved
- each stock has persistent features and action labels by run date
- SEC filing capture is integrated for current watchlist names
- the system can explain its recommendation using stored inputs and derived features

## Open Questions

- Does `FMP Starter` provide enough earnings and fundamentals depth for the exact watchlist we care about?
- Should raw FMP payloads be stored in Postgres or in object storage from day one?
- Is the first delivery artifact better as markdown, HTML, or a minimal app page?

## Recommended Build Order

1. define normalized schemas
2. build watchlist store
3. build FMP price and benchmark ingestion
4. build SEC filings ingestion
5. build feature computation
6. build weekly scoring logic
7. build weekly report output
8. add daily digest
